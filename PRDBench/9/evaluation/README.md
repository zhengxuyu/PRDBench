# Business Travel Intelligent Planning System - Test Plan

## Overview

This test plan is based on the PRD document and evaluation metrics, providing a complete automated testing framework for the Business Travel Intelligent Planning System. The test plan contains 42 test points, covering all core functional modules of the system.


## Test Types

### 1. Shell Interaction Tests (shell_interaction)
- **Quantity**: 40 test points
- **Purpose**: Test functions that require real user interaction with the command line
- **Execution Method**: Simulate user operations through standard input
- **Examples**: Program startup, menu navigation, data input validation, etc.

### 2. Unit Tests (unit_test)
- **Quantity**: 1 test point
- **Purpose**: Test functions that can be verified by directly calling source code functions
- **Execution Method**: Using pytest framework
- **Examples**: File export functionality verification

### 3. File Comparison Tests (file_comparison)
- **Quantity**: 1 test point
- **Purpose**: Verify whether the output files generated by the program are correct
- **Execution Method**: Compare with expected output files
- **Examples**: Markdown format export verification

## Test Coverage

### Core Functional Modules
1. **Program Startup and Interface** (2 test points)
   - Program startup verification
   - Create new trip functionality

2. **Trip Information Management** (8 test points)
   - City input validation (valid/invalid)
   - Stay duration settings (integer/decimal hours)
   - Time window constraint settings
   - Trip template save/load

3. **Transportation Plan Generation** (8 test points)
   - Transportation data display (high-speed rail/flights)
   - Optimization objective selection (time/cost/balanced)
   - Transportation preference settings
   - Transfer limit restrictions

4. **Conflict Detection** (6 test points)
   - Basic conflict detection functionality
   - Hard conflict identification (rail to flight/flight to rail transfers)
   - Soft conflict identification (insufficient rest time)
   - Conflict resolution suggestions

5. **Decision Analysis** (3 test points)
   - Multi-attribute evaluation
   - Quantitative scoring
   - Solution priority ranking

6. **Export Functionality** (6 test points)
   - Markdown format export
   - File save verification
   - Trip summary generation (distance/time cost/key nodes)

7. **User Interface** (6 test points)
   - Menu system (hierarchical structure/return navigation)
   - Input validation (numbers/menu selection)
   - Date format support (standard/relative formats)

8. **System Quality** (3 test points)
   - Trip save/load functionality
   - System stability testing
   - User experience friendliness

## Execution Methods

### 1. Manual Execution of Individual Tests
```bash
# Shell interaction test
python src/main.py < evaluation/inputs/startup_test.in

# Unit test
pytest evaluation/tests/test_file_export.py::test_file_save_verification

# File comparison test
python src/main.py < evaluation/inputs/markdown_export_test.in
# Then compare the generated file with expected_outputs/expected_trip_export.md
```

### 2. Batch Test Execution
```bash
# Execute all unit tests
pytest evaluation/tests/

# Execute specific types of tests
for test_file in evaluation/inputs/*.in; do
    echo "Executing test: $test_file"
    python src/main.py < "$test_file"
done
```

## Scoring Standards

Each test point uses a 3-level scoring system:
- **2 points**: Fully meets requirements, functionality works normally
- **1 point**: Basically meets requirements, but has minor issues
- **0 points**: Does not meet requirements or functionality is unavailable

Total score calculation: Score of each test point × corresponding weight, then sum

## Test Data Description

### Input File Format

- Arranged in the order of program interaction flow
- Includes various boundary cases and exceptional inputs

### Expected Output
- Standard output format defined based on PRD requirements
- Contains necessary functionality verification points
- Considers user experience and interface friendliness
